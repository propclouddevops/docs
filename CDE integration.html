Content Distribution Engine (CDE) Integration

Overview

The Content Distribution Engine (CDE) is a content processing and distribution pipeline that is part of the Ringier Event Bus ecosystem. The Event Bus is a group-wide, event-driven messaging infrastructure that transfers events from senders (nodes) to receivers. CDE is a downstream service that consumes events from the bus to redistribute content to partner sites, generate video content, and provide curated recommendations.

Our platform is registered as a Node on the bus — both emitting events (listings, articles, leads) and receiving data back (videos, sidebar links, listing reconciliation feeds).

The system has two directions:

  • Outbound: We push property data as events to the bus via an AWS SQS FIFO queue. CDE and other services consume these events.
  • Inbound: CDE publishes video posts back onto our listings, provides sidebar link recommendations, and exposes feeds we use for reconciliation and homepage recommendations.

Outbound Flow: How Content Gets Distributed

The Pending Queue

When a listing is created or modified anywhere in the system, a pending flag is set on that listing:

  • Flag = 1 (Create): Listing is brand new
  • Flag = 2 (Update): Listing was modified

This flag acts as a lightweight work queue — no separate queue table is needed.

flowchart LR A[New listing created] --> PQ[cde_pending = 1 or 2] B[Listing edited in CRM] --> PQ C[Mandate status changed] --> PQ D[Photos added/removed] --> PQ E[Moderation approved] --> PQ F[Priority ad updated] --> PQ G[Remote sync updated] --> PQ PQ --> HF[Hangfire Job] HF --> SR[Send to SQS] SR --> CL[Clear flag on success]

Processing Cycle

Every 30 minutes, a Hangfire job:

  1. Queries all listings with a pending flag
  2. Sends each one to the SQS FIFO queue as a JSON event
  3. Clears the flag only for successfully sent listings
  4. Failed listings remain in the queue and are automatically retried on the next cycle

Event Payload

Each event sent to SQS contains:

  • Event type (e.g. ListingCreated, ListingUpdated, ArticleCreated)
  • Node ID identifying the source platform
  • Unique reference for deduplication
  • Full payload with all listing/article/lead data

Content Types

Event Envelope (v2.0.0)

All events sent to the bus follow the v2.0.0 payload structure:

  • events — array of event name strings (e.g. ["ListingCreated"])
  • from — our Node ID (configured per content type: one for listings/leads, one for articles)
  • reference — unique GUID per event instance
  • created_at — ISO 8601 timestamp with timezone
  • payload — the full content payload (structure defined in the Marketplace Events spreadsheet)
  • version — hardcoded "2.0.0"
  • route — empty string (delivered to all subscribed nodes)

Listings

The most comprehensive payload, including:

  • Listing status (online / offline — based on mandate status, moderation, site state)
  • Prominence level (top, gold, silver, standard)
  • All property attributes (bedrooms, bathrooms, area, amenities, etc.)
  • All images in multiple sizes
  • Video URL if available
  • Full location hierarchy (province → town → suburb)
  • All text fields in both English and French

Articles (Blog Posts)

Sent for portal sites only (requires both CdeEnabled and isPortal to be true):

  • Publication status
  • Images in multiple sizes
  • Full multilingual content

Leads

A minimal payload containing the lead reference, type, and associated listing reference.

Inbound Flows: What CDE Sends Back

flowchart TB VF[Video Feed - JSON] -->|Hourly| V[Video URLs on Listings] SL[Sidebar Links - JSON] -->|Weekly| S[Sidebar Link Widgets] OL[Online Listings - XML Feed] -->|Every 30 min| R[Reconciliation: Force Offline] HL[Homepage Listings - XML Feed] -->|On page load| H[Homepage Recommendations]

Video Posts

CDE creates and publishes property videos. An hourly task fetches a JSON feed of recently posted videos and writes the video URLs back to the corresponding listings.

Sidebar Recommended Links

A weekly task fetches a JSON feed of recommended internal links from an S3 bucket. These are stored locally and displayed as sidebar widgets on relevant pages, providing CDE-curated cross-linking.

Online Listing Reconciliation

Every 30 minutes, the platform fetches an XML feed of all listings CDE considers "online". Any listing in that feed that is not legitimately active in our system (e.g. expired, deleted, unmoderated) gets flagged for an update, which causes the next processing cycle to send an "offline" status event — ensuring CDE takes down stale listings.

Homepage Listing Recommendations

CDE provides curated listing feeds (separate for English and French) that are consumed on the homepage to display recommended listings.

Bulk Operations (Admin)

Administrators can trigger bulk operations from the admin panel:

  • Bulk send all listings — re-queues all active listings for distribution (optionally filtered by site)
  • Bulk send by references — re-queues specific listings by reference number
  • Bulk send recent — re-queues all listings updated within the last N days
  • Bulk send articles — sends all blog posts (with a configurable delay between each)
  • Bulk send leads — sends all leads from the past year
  • Send test event — sends a single listing or article for testing

Error Handling

  • Per-listing retry: If a listing fails to send, its pending flag remains set. It will be retried on the next 30-minute cycle automatically.
  • Concurrent execution guards: All tasks prevent overlapping runs to avoid duplicate processing.
  • Master kill switch: A global CdeEnabled setting gates all outbound events. Disabling it immediately stops all distribution without removing pending flags. (Note: the admin "Send test event" action bypasses this check.)

Configuration

The integration is configured entirely through admin settings:

  • Enable/Disable: Master on/off toggle
  • AWS Credentials: SQS queue URL, access key, and secret key
  • Node IDs: Separate identifiers for listings and blog content
  • Feed URLs: Configurable endpoints for all inbound feeds (video, sidebar, reconciliation, homepage)
  • Throttling: Configurable delay between messages during bulk operations
